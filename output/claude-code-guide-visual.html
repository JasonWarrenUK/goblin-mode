Section 01

Configuration
Scopes

Every customisation lives at one of four levels. The more specific level wins when two conflict.

Precedence — highest wins
Managed  — IT-deployed, all users system-level
Local  — personal project override .claude/settings.local.json
Project  — shared with team .claude/settings.json
User  — all your projects ~/.claude/settings.json
Arrow of precedence: Managed › Local › Project › User
Scope reference table
Scope Shared? Use for
Managed All users Org-wide policies
Local Just you Personal project tweaks
Project Committed to git Team-wide standards
User Just you Cross-project preferences
The same pattern applies to
CLAUDE.md files Subagents Skills Rules

A ~/.claude/ version is personal to you. A .claude/ version is shared with the team.

Section 02

CLAUDE.md
Files

Plain markdown Claude reads at the start of every session. Specific, concrete instructions. Not enforced rules — Claude treats them as instructions.

Load locations
PathLoaded when
~/.claude/CLAUDE.md Every project
CLAUDE.md This project only
.claude/CLAUDE.md This project only
subdir/CLAUDE.md When Claude reads that dir
💡 Run /init — Claude generates a starter CLAUDE.md by analysing your codebase.
✓ Effective
# Project: payments-api ## Build & test - Run tests: `npm test` - Lint: `npm run lint` ## Conventions - Use 2-space indentation - Handlers in `src/handlers/` - Validate inputs with Zod - Return `{ error, code }`

Commands are exact and runnable. Every rule is verifiable. Claude can check "did I follow this?" for each line.

✗ Weak
# Project notes Make sure the code is well organised and follows best practices. Tests are important. Keep things clean and consistent.

None of these are testable. "Well organised" means different things to different people. Claude makes its own interpretation every time.

The key principle

For each instruction, Claude should be able to answer: "Can I verify whether I followed this?" If not, the instruction is too vague.

Section 03

Rules

CLAUDE.md extended. Markdown files in .claude/rules/ — each file covers one topic. Path-scoped rules only load when Claude touches matching files.

Directory structure
.claude/ ├── CLAUDE.md └── rules/ ├── testing.md # loads every session ├── api-design.md # loads every session └── frontend.md # can be path-scoped
Why path scoping matters

Rules without path scoping load at session start — same as CLAUDE.md.
Rules with path scoping only appear in context when relevant, keeping the context window lean.

✓ Effective path-scoped rule
--- paths: - "src/api/**/*.ts" --- # API Rules - All endpoints must validate inputs with Zod - Return `{ data, error }` shape consistently - Include JSDoc on exported handler functions

Path scope is correct, content is actionable. Only loads when working on API files.

✗ Correct scope, weak content
--- paths: - "src/api/**/*.ts" --- When writing API code, try to follow good API design principles and make sure things work correctly.

Correct scoping doesn't compensate for vague instructions. Path scope is doing all the work here.

How path scoping triggers

When Claude reads, writes, or edits any file matching the glob pattern, the rule loads into context. Unrelated work stays clean.

Section 04

Skills

Reusable instruction sets in SKILL.md files. Directory name becomes the /slash-command.

Two skill types
Command Skill

disable-model-invocation: true — you trigger with /name. Good for actions with side effects like /deploy or /commit.

Knowledge Skill

user-invocable: false — Claude loads it automatically when relevant. Good for reference material or domain guidelines.

Key frontmatter fields
FieldPurpose
nameSlash command name (defaults to dir name)
descriptionWhen Claude should load this skill — front-load the use case
disable-model-invocationOnly trigger manually with /name
user-invocableSet false to hide from / menu
allowed-toolsRestrict available tools while skill runs
argument-hintAutocomplete hint, e.g. [issue-number]
contextSet fork to run in isolated subagent
✓ Effective — command skill
--- name: create-component description: Scaffold a new React component with tests disable-model-invocation: true argument-hint: <ComponentName> allowed-tools: Write, Read --- Create a new React component named $ARGUMENTS. 1. Create `src/components/$ARGUMENTS/$ARGUMENTS.tsx` 2. Create `src/components/$ARGUMENTS/$ARGUMENTS.test.tsx` with basic render test 3. Export from `src/components/index.ts` Follow existing patterns in `src/components/`.
✗ Weak — vague, unsafe
--- name: create-component description: Makes components --- # no disable-model-invocation! Create a component for the user. Make it look nice and work properly.

Without disable-model-invocation, Claude might scaffold a component when you only asked a question. Vague description triggers on any UI query.

Storage locations
~/.claude/skills/<name>/SKILL.md all projects
.claude/skills/<name>/SKILL.md this project

Section 05

Subagents

Specialised AI assistants that run in their own context window. Claude delegates tasks automatically based on description.

Built-in agents
Explore Read-only, fast codebase search
Plan Used during plan mode
general-purpose Fallback for unmatched tasks
Key frontmatter fields
FieldPurpose
nameIdentifier (lowercase, hyphens)
descriptionWhen Claude should delegate here
toolsAllowlist of tools
disallowedToolsDenylist (applied before tools)
modelsonnet, opus, haiku, or inherit
maxTurnsMax agentic turns before stopping
✓ Effective — focused agent
--- name: security-reviewer description: Reviews code for security vulnerabilities. Use when reviewing authentication, authorization, input handling, database queries, or code that handles user data. tools: Read, Grep, Glob model: sonnet --- You are a security code reviewer. When invoked: 1. Read the changed files 2. Check for: SQL injection, XSS, IDOR, missing auth checks, secrets in code 3. Report: file, line, severity, concrete fix Do not suggest unrelated refactors. Stay focused.
✗ Weak — catches everything, costs everything
--- name: helper description: Helps with coding tasks tools: Read, Write, Bash, Edit, Grep, Glob model: opus --- You are a helpful coding assistant. Help the user with whatever they need.

Description matches everything — Claude delegates constantly. Opus for generic help wastes money. Full tool access including Bash is a security risk with no focused purpose.

The agent body is the entire system prompt

It receives only that — not the full Claude Code system prompt. Specificity in the description is what routes tasks correctly.

Section 06

Hooks

Shell commands triggered by lifecycle events. Enforce rules, auto-format, log activity, or block unsafe operations.

Lifecycle events
SessionStart
Session begins
PreToolUse
Before Claude uses a tool
PostToolUse
After a tool runs
Stop
When Claude finishes responding
UserPromptSubmit
When you submit a message

To block an action: exit code 2 + print reason to stderr.

Hook configuration shape
// in settings.json { "hooks": { "PostToolUse": [ { "matcher": "Write|Edit", "hooks": [ { "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/lint.sh" } ] } ] } }
✓ Auto-lint after edits
#!/bin/bash FILE=$(cat | jq -r '.tool_input.file_path') if [[ "$FILE" == *.js || "$FILE" == *.ts ]]; then npx eslint --fix "$FILE" 2>/dev/null fi exit 0

Matcher specific. Parses stdin for file path. JS/TS only. Exit 0 = non-blocking.

✗ Empty matcher catastrophe
{ "PreToolUse": [{ "matcher": "", "hooks": [{ "command": "echo 'doing something' >> /tmp/log" }] }] }

Empty matcher fires on every tool call — Read, Grep, Glob — thousands of times per session. Spawns a shell each time.

Section 07

MCP
Servers

Model Context Protocol. Gives Claude access to external tools: databases, APIs, issue trackers, browsers.

Three transport types
🌐
HTTP (recommended)
Cloud services. Stable, well-supported.
claude mcp add --transport http <name> <url>
⚙️
stdio
Local process on your machine.
claude mcp add --transport stdio <name> -- <cmd> [args]
⚠️
SSE (deprecated)
Prefer HTTP. Being phased out.
claude mcp add --transport sse <name> <url>
Management commands
claude mcp list # all servers claude mcp get <name> # one server details claude mcp remove <name># remove a server
Scope flags
FlagStored inShared?
--scope local ~/.claude.json (per-project) No
--scope project .mcp.json in project root Yes — commit it
--scope user ~/.claude.json No — all projects
✓ Adding GitHub (correct way)
# Add once — user scope, all projects claude mcp add \ --transport http \ --scope user \ github \ https://api.githubcopilot.com/mcp/

Then run /mcp inside Claude Code, select GitHub, authenticate with OAuth.

✗ Never edit ~/.claude.json directly

~/.claude.json is managed by Claude Code and may be overwritten. Absolute paths won't work on another machine or in CI. Use claude mcp add — it handles the correct format and location.

Inside Claude Code
/mcp # view status, authenticate OAuth

Section 08

Plugins

Package skills, subagents, hooks, and MCP servers together. Share, version, and install across projects. Skills are namespaced to prevent conflicts: /my-plugin:deploy

Plugin structure
my-plugin/ ├── .claude-plugin/ │ └── plugin.json ← manifest (only file here) ├── skills/ │ └── deploy/ │ └── SKILL.md ├── agents/ │ └── reviewer.md └── hooks/ └── hooks.json
Test and distribute
# Test locally claude --plugin-dir ./my-plugin # Reload without restarting /reload-plugins # Teams install from GitHub /plugin install owner/repo
✓ Effective plugin.json
{ "name": "acme-standards", "description": "Acme Corp coding standards — linting hooks, review agents, commit conventions", "version": "1.2.0", "author": { "name": "Acme Platform Team" }, "homepage": "https://github.com/acme/claude-standards" }

Clear description, semver, homepage lets teammates find the source.

✗ Weak plugin.json
{ "name": "MyPlugin", ← PascalCase breaks namespacing "description": "my plugin", "version": "1" ← not semver }

Plugin names must be lowercase with hyphens — PascalCase breaks skill namespacing. Version must be semver. These are technically valid JSON but cause behavioural problems.

The key difference from standalone config

Plugin skills are namespaced: /plugin-name:skill-name. This prevents conflicts when multiple plugins provide similarly-named skills.

Section 09

Worktrees

Run multiple Claude sessions simultaneously, each in its own isolated copy of the repository, on separate branches, without interference.

Advanced topic

Come back to this once you're comfortable with skills and subagents. The machinery behind this is git worktrees — Claude wraps the workflow.

Basic usage
# Start Claude in an isolated worktree claude --worktree feature-name

Creates .claude/worktrees/feature-name/ on a new branch called worktree-feature-name. Cleans up automatically if no changes were made.

Subagent worktree isolation
--- name: my-agent isolation: worktree ---
Parallel sessions diagram
Main repo  /  branch: main
Your normal Claude Code session — unaffected
Worktree 1  /  branch: worktree-auth-rewrite
Claude session — isolated, separate branch
Worktree 2  /  branch: worktree-new-api
Claude session — isolated, separate branch
Worktree safety notes
→ Check which branches exist before creating new ones
→ Never remove a worktree while your shell is inside it
→ After removing, cd back to the main repo directory
